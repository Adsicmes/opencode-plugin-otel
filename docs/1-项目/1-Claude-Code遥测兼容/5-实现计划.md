# Claude Code 遥测兼容实现计划

## 交付范围

实现显式 `claude-code` Profile，覆盖五类核心 Metrics、五类核心 Events、三类核心 Span、交互关联和默认内容脱敏；保留默认 `opencode` Profile 的现有行为。

## 阶段与任务

| 阶段 | 任务 | 主要文件 | 验证方式 | 完成条件 |
| --- | --- | --- | --- | --- |
| 1. 契约基线 | 将调研结果整理为机器可断言的兼容矩阵和 fixture | `tests/fixtures/`、参考文档 | 评审字段、类型、单位和枚举 | MVP 每个信号都有 expected contract |
| 2. Profile 配置 | 增加 `telemetryProfile`、环境变量、option、类型和优先级 | `src/config.ts`、`src/types.ts`、`tests/config.test.ts` | 配置单元测试 | 默认名称、配置和计数语义不变，两个合法 Profile 可选 |
| 3. Schema 模块 | 集中定义 Metric/Event/Span 名称、映射和内容策略 | 新增 `src/schema.ts` 或同职责模块 | 纯函数单元测试 | handlers 无需自行拼接 Profile 名称 |
| 4. 隐私修复 | 默认移除 tool input/output/parameters 明文，统一 prompt/response redaction | `src/handlers/message.ts`、相关测试 | 敏感字段否定断言 | 默认 exporter 输出无正文和工具内容 |
| 5. 关联上下文 | 增加 prompt ID、event sequence、tool use ID 和 session cleanup | `src/types.ts`、`src/util.ts`、handlers | 生命周期和容量测试 | 同 turn 事件关联一致，结束后状态释放 |
| 6. Events | 引入统一 event emitter，对齐五类核心事件 | `src/index.ts`、`src/handlers/*.ts` | LogRecord 契约测试 | body、属性、时间和 correlation 符合基线 |
| 7. Metrics | 按 Profile 创建和记录标准/扩展 instruments | `src/otel.ts`、`src/handlers/*.ts` | Metric reader 测试 | 名称、类型、unit 和属性符合基线，无重复累计 |
| 8. Traces | 对齐 interaction、llm_request、tool 名称和核心属性 | `src/handlers/session.ts`、`src/handlers/message.ts` | in-memory exporter 测试 | 父子层级、状态和属性符合基线 |
| 9. 端到端 | 经 Collector 和 SigNoz 验证真实 OTLP | 测试配置、验证报告 | 查询和截图/导出证据 | MVP Dashboard 与关联查询可用 |
| 10. 文档发布 | 更新 README、配置参考、兼容矩阵和变更记录 | `README.md`、`docs/`、`CHANGELOG.md` | 链接和 docctl 校验 | 使用者可启用、验证、回退 |

## 详细实施顺序

### 1. 固化兼容基线

- 从 Claude Code 参考文档提取 MVP 字段表。
- 给每个字段标记 `exact`、`approximate`、`unsupported` 或 `extension`。
- 为 Metric、Event 和 Span 分别建立最小 fixture。
- 明确 baseline 版本或访问日期，避免测试追随上游页面自动漂移。

### 2. 配置与 Profile

- 扩展 `PluginConfig` 和 `OtelPluginOptions`。
- 新增 `OPENCODE_TELEMETRY_PROFILE`。
- 保持 option → env → default 优先级。
- 启动诊断日志增加 Profile。
- 明确 Profile 与 `metricPrefix`、disabled metrics/traces 的组合规则。
- Claude Code Profile 默认 temporality 为 `delta`，显式覆盖优先并报告兼容性降级。

### 3. 内容安全前置改造

- 删除默认 tool Span 的 `TOOL_PARAMETERS` 和明文 `INPUT_VALUE`。
- 将成功输出和错误明文替换为大小、状态与有界摘要。
- 检查 subtask description、permission title 和 command argument 是否可能携带内容。
- 增加跨 Logs/Traces 的敏感内容否定测试。

### 4. 关联模型

- 在 user turn 建立 prompt context。
- event emitter 统一分配 sequence 和 timestamp。
- 扩展 pending tool 状态保存 call ID 与相关 OTel context。
- 在 LLM/tool Span 结束前保存 `SpanContext`，通过 `LogRecord.context` 关联后发出的 Event。
- permission 没有可靠 call ID 时不与 tool result 强行关联。
- 在 idle/error 清理全部新增状态。
- 对乱序、缺失 running、缺失 permission 等场景增加测试。

### 5. 信号映射

- 先 Events，再 Metrics，最后 Traces；每一步独立运行默认 Profile 回归测试。
- 不在多个 handler 重复构造公共事件字段。
- reasoning token 和 opencode-only signals 保持独立，不塞入 Claude Code 标准字段。
- 保留 OpenInference 扩展属性时，确认不会携带默认禁止的内容。

### 6. 集成验证

- 启动本地 Collector，保存规范化 OTLP 输出。
- 执行成功、失败、工具、权限、diff、commit 和 subagent 场景。
- 导入目标 Claude Code Dashboard 或执行等价查询。
- 验证 Resource 仍标识 opencode producer，Schema 标记可过滤。
- 记录无法兼容和语义近似项。

## 文件变更清单

预计修改：

- `src/config.ts`：Profile 配置和优先级。
- `src/types.ts`：Profile、关联状态和 instrument 类型。
- `src/otel.ts`：按 Profile 创建 instruments、真实 Resource 和 scope。
- `src/index.ts`：初始化 Profile、统一 emitter 和上下文。
- `src/handlers/session.ts`：interaction、session lifecycle 和 cleanup。
- `src/handlers/message.ts`：API、tool、token/cost、内容门控和 Span 映射。
- `src/handlers/permission.ts`：decision 枚举和 tool correlation。
- `src/handlers/activity.ts`：lines/commit Profile 映射。
- `src/util.ts`：公共关联和映射 helper，仅在确有复用时增加。
- `tests/`：配置、契约、隐私、关联、cleanup 和回归测试。
- `README.md`、`docs/`、`CHANGELOG.md`：配置、矩阵、迁移和发布说明。

## 可并行工作

- Profile 配置测试与兼容矩阵 fixture 可并行。
- Metrics 映射与 Event 契约测试可在 Schema 模块确定后并行。
- README 草稿可与端到端 Collector 验证并行，但最终示例必须以实测配置为准。

Trace 改造依赖 prompt/tool 关联模型，不应提前并行实现。

## 验证计划

每个阶段至少运行：

```bash
bun run typecheck
bun test
```

最终增加：

- 默认 Profile 全量回归。
- Claude Code Profile 契约测试。
- 敏感内容否定测试。
- Collector OTLP 捕获验证。
- SigNoz Dashboard、Logs 和 Traces 查询验证。
- `docctl rebuild-index`、`docctl validate` 和 `git diff --check`。

## 发布与回滚

- 作为 opt-in 能力发布，不改变默认 Profile。
- Release notes 明确“Schema-compatible telemetry”，不得描述为官方 Claude Code telemetry。
- 使用者通过配置切回 `opencode` Profile并重启即可回滚。
- 若发现内容泄露风险，立即禁用 Logs/Traces 或整个 telemetry，并在修复前撤回兼容 Profile。

## 风险检查

| 风险 | 缓解措施 |
| --- | --- |
| 名称兼容但语义不兼容 | 兼容矩阵与 exact/approximate 标记；Dashboard 实测 |
| reasoning token 重复或误分类 | 不映射到官方四类，单一累计路径测试 |
| 关联 Map 泄漏 | bounded map、session cleanup 和容量测试 |
| 默认工具内容泄露 | 隐私修复作为信号映射前置门槛 |
| 上游 Claude Code Schema 漂移 | 固定基线，升级走显式评审和契约变更 |
| 消费者误认 producer | 保留真实 Resource/scope，增加 Schema 兼容标记 |
| Profile 条件散落 | 集中 Schema 模块与统一 event emitter |

## 完成门槛

- [验收标准](3-验收标准.md)全部满足。
- 默认 Profile 的名称、配置和计数语义无回归，工具内容移除已作为安全性行为变化记录。
- Claude Code Profile 的隐私、契约和关联测试通过。
- SigNoz 实测完成并形成验证报告。
- README、兼容矩阵和 CHANGELOG 已同步。
