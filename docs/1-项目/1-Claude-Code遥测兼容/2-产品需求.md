# Claude Code 遥测兼容产品需求

## 背景

本插件当前以 opencode 语义输出 OpenTelemetry。设置 `OPENCODE_METRIC_PREFIX=claude_code.` 可以改变 Metric 名称前缀，但不能使整体数据契约兼容 Claude Code：

- Log body 当前是 `user_prompt`、`api_request` 等，Claude Code 使用 `claude_code.user_prompt`、`claude_code.api_request`。
- Trace 当前使用 `session`、`llm`、`tool.<工具名>`，Claude Code 使用 `interaction`、`llm_request`、`tool` 及固定子 Span。
- 当前缺少 `prompt.id`、`event.sequence`、`tool_use_id` 等关联字段。
- 部分属性名、枚举、单位和 Span kind 与 Claude Code Schema 不一致。
- 工具输入和输出目前可能直接进入 Span，默认隐私边界弱于 Claude Code 的默认脱敏行为。
- opencode 特有指标与 Claude Code 指标混用时会增加 Dashboard 适配成本。

使用者希望复用现有 Claude Code SigNoz Dashboard、查询和告警，而不是为 opencode 维护一套平行资产。

## 目标

1. 提供一个显式、可测试、可回退的 Claude Code 遥测兼容 Profile。
2. 让核心使用量、成本、会话、代码改动、工具调用和错误数据可直接被 Claude Code Dashboard 查询。
3. 让一次用户提示产生的日志和 Span 可以通过稳定关联键还原为同一交互。
4. 默认不上传提示正文、助手回答、工具参数、工具输入输出或原始 API body。
5. 保留数据生产者的真实身份和来源，避免把 opencode 遥测误报为 Anthropic 官方客户端数据。

## 用户与场景

### 目标用户

- 已经部署 SigNoz 或其他 OTLP 后端的 opencode 使用者。
- 已有 Claude Code Dashboard、告警和成本查询的可观测性团队。
- 需要比较 opencode 与 Claude Code 使用情况的平台工程团队。
- 需要审查遥测隐私和数据最小化策略的安全团队。

### 核心场景

1. 集成者启用兼容 Profile 后，现有 Claude Code Token、成本和会话图表可以显示 opencode 数据。
2. 运维人员从 `claude_code.user_prompt` 事件按 `prompt.id` 找到对应 API 请求、工具执行和 Trace。
3. 维护者在不改变默认 Profile 的前提下升级插件，并可随时关闭兼容 Profile 回退到原 Schema。
4. 安全团队能证明兼容 Profile 默认只发送结构化元数据，不发送代码和对话正文。

## 功能需求

### FR-1 显式兼容 Profile

- 新增 Profile 配置，至少支持 `opencode` 和 `claude-code`。
- 未配置时必须使用 `opencode`。除“默认移除当前工具输入输出明文”这一明确的安全加固外，现有名称、配置、计数与关联语义保持不变。
- 插件 tuple option 优先于 `OPENCODE_*` 环境变量，环境变量优先于默认值。
- Profile 必须统一控制 Metrics、Logs 和 Traces，避免只改变名称前缀形成“部分兼容”。
- `metricPrefix` 等现有高级配置与 Profile 冲突时，行为必须确定并有文档说明。

### FR-2 Metric 兼容

MVP 必须输出以下 Claude Code Metric：

| Metric | MVP 要求 |
| --- | --- |
| `claude_code.session.count` | 每个真实 opencode 会话开始时增加；包含可证明的启动/子代理维度 |
| `claude_code.token.usage` | 输出 `input`、`output`、`cacheRead`、`cacheCreation`；opencode 独有 reasoning 不得冒充官方四类 Token |
| `claude_code.cost.usage` | 每个已完成 assistant message 增加真实估算成本 |
| `claude_code.lines_of_code.count` | 输出 `type=added/removed`，并明确 opencode diff 的 gross positive churn 语义差异 |
| `claude_code.commit.count` | 仅在插件确实检测到 Git commit 时增加 |

- MVP 不要求 `pull_request.count`、`code_edit_tool.decision` 和 `active_time.total`，除非实现时已有可靠事件源。
- Claude Code Profile 默认不得以 `claude_code.*` 名称输出 opencode 独有 Metric。
- 无法一一映射的 opencode 扩展 Metric 可以关闭，或继续使用 `opencode.*` 命名空间，但不得污染 Claude Code 命名空间。
- Metric 单位、类型、属性名和枚举必须纳入契约测试。
- Claude Code Profile 默认使用 `delta` temporality；使用者显式覆盖 temporality 时允许发送，但必须记录兼容性降级。

### FR-3 Event 兼容

MVP 必须输出：

- `claude_code.user_prompt`
- `claude_code.api_request`
- `claude_code.api_error`
- `claude_code.tool_result`
- `claude_code.tool_decision`

每条事件必须满足：

- Log body 使用完整的 `claude_code.<event>` 名称。
- `event.name` 使用不带前缀的事件名。
- 包含 ISO 8601 `event.timestamp` 和会话内单调递增的 `event.sequence`。
- 包含真实 `session.id`；属于用户交互时包含同一 `prompt.id`。
- API 事件使用可获得的模型、成本、耗时和 Token 字段。
- `tool_result` 与 tool Span 使用 opencode `callID` 作为 `tool_use_id`。permission 事件优先使用上游 `callID`；缺失时生成随机 UUID 作为不透明关联值，不能用 permission ID 冒充。
- 权限 `source` 仅映射有确定语义的值：`allowAlways` 映射为 `user_permanent`，一次性 `allow` 映射为 `user_temporary`，明确拒绝映射为 `user_reject`；无法区分 abort、配置或 hook 时不得猜测。
- 不产生没有真实 opencode 事件源的认证、MCP、插件、Skill、Hook 或反馈事件。

### FR-4 Trace 兼容

MVP 必须形成以下层级：

```text
claude_code.interaction
├── claude_code.llm_request
└── claude_code.tool
```

- 一个用户 turn 对应一个 `interaction` Span。
- assistant model 请求对应 `llm_request` 子 Span。
- 工具调用对应 `tool` 子 Span，名称不能拼接具体工具名。
- Span 必须包含 `span.type`、`session.id` 和各类型的核心属性。
- 失败的 `llm_request` 和工具执行必须设置 OTel `ERROR` 状态。
- LogRecord 在存在对应 Span context 时应携带 `trace_id` 和 `span_id`；不存在时仍可通过 `session.id`、`prompt.id` 和 `tool_use_id` 关联。
- subagent 的 Span 应嵌套在触发它的工具或交互下；无法可靠建立父子关系时不得构造虚假父级。

### FR-5 隐私门控

- 默认将提示正文、助手回答、工具参数、工具输入和工具输出排除或标记为 `<REDACTED>`。
- 兼容 Profile 首次交付不提供原始 API body 上传能力。
- 如果提供内容开关，必须使用 `OPENCODE_*` 前缀，并逐项控制 prompt、response、tool details 和 tool content。
- 内容开关不得隐式开启其他内容类别。
- 错误字段要避免直接上传可能包含命令、路径或正文的原始异常；默认使用有界摘要。
- Resource 必须保留真实 `service.name`，不得默认为 `claude-code`。
- 不得生成或复制虚假的 `user.email`、`user.account_id`、`user.account_uuid`、`organization.id`、Claude Code `service.version` 或 Anthropic 官方 scope。

### FR-6 生产者身份与来源

- 默认 `service.name` 保持 `opencode`，允许使用者通过现有 Resource 配置显式覆盖。
- instrumentation scope 保持本项目命名空间，例如 `com.opencode`，不得使用 `com.anthropic.claude_code.*`。
- 增加可查询的兼容标记，例如 `telemetry.schema.name=claude-code` 和目标 Schema 版本/日期。
- Claude Code Profile 使用 `service.version` 和 instrumentation scope version 上报插件真实版本，不得把插件版本放入语义上表示 Claude Code 版本的 `app.version`。

### FR-7 可观测与诊断

- 启动日志必须记录当前 Profile，但不能记录 OTLP headers 或内容数据。
- 配置冲突、无效 Profile 和不能映射的字段必须有确定行为。
- 插件处理单个遥测事件失败不得中断 opencode 主流程。
- Profile 切换后的事件、Metric 和 Span 数量可通过测试 exporter 或 Collector 验证。

### FR-8 文档与兼容矩阵

- README 必须说明如何启用、关闭和验证 Claude Code Profile。
- 参考文档必须维护“完全兼容、语义近似、未实现、不可映射”四类矩阵。
- 每个近似映射必须说明差异，尤其是 lines of code、reasoning tokens、权限 source 和 session start type。
- 兼容目标要绑定到一个明确的 Claude Code 文档访问日期或 Schema 基线，不承诺自动跟随最新版本。

## 非功能需求

### 兼容性

- 默认 Profile 的 Metric、Event 和 Trace 名称不得变化；默认工具内容移除属于已声明的安全加固，不计为 Schema 回归。
- 现有配置键继续有效；新增 option 与 `PluginConfig` 字段保持 1:1。
- 不引入 `@opentelemetry/sdk-node`。

### 性能与资源

- 每个会话新增的关联状态必须有界。
- session idle 清理当前执行状态，但保留 session sequence 和晚到事件所需的有界关联；session deleted/error 时清理全部关联状态。
- `pendingToolSpans` 和 `pendingPermissions` 继续使用 `setBoundedMap`。
- 不为兼容 Profile 重复发送同一逻辑信号。

### 安全

- 默认配置通过代码和测试证明不发送内容字段。
- 示例不得包含真实 endpoint 密钥、邮箱、账号或组织标识。
- Compatibility 不得改变 OTLP 鉴权和动态 headers 的安全边界。

### 可维护性

- Schema 映射集中管理，handlers 不应散落大量 Profile 条件分支。
- 契约测试应比较稳定的规范化对象，不依赖随机 ID、绝对时间或 exporter 批次顺序。

## 非目标

- 让 Anthropic、SigNoz 或其他系统误认为数据来自官方 Claude Code 客户端。
- 绕过许可证、计费、访问控制、审计或产品识别机制。
- 伪造用户、账号、组织、终端、Claude Code 版本或官方 instrumentation scope。
- 在 MVP 中实现 Claude Code 当前所有事件和 Beta 详细 tracing。
- 对 opencode 不提供的数据进行推测填充。
- 保证所有第三方 Claude Code Dashboard 无修改兼容；只对文档化字段和已验证 Dashboard 建立承诺。

## 依赖与约束

- opencode SDK 必须提供 session、message、tool、permission 和 diff 的真实事件源。
- 部分 Claude Code 字段没有 opencode 等价物，必须允许缺省。
- Claude Code Schema 会演进，兼容矩阵和契约 fixture 需要版本化维护。
- Metrics Token/成本继续只在 `message.updated` 中累计，不能在 step-finish 或其他路径重复增加。

## 优先级

### MVP

- Profile 配置与默认行为保护。
- 五类核心 Metrics 映射。
- 五类核心 Events 与关联字段。
- `interaction`、`llm_request`、`tool` Trace。
- 默认内容脱敏修复。
- 契约测试、README 和 SigNoz 验证。

### 后续

- assistant response 事件。
- permission wait 和 tool execution 子 Span。
- PR、active time 和 edit decision Metric。
- 具有真实事件源的 MCP、Skill、Plugin、Hook 事件。
- 细粒度内容采集开关。

## 成功指标

- Claude Code Profile 的契约测试全部通过。
- 默认 Profile 的既有测试全部通过。
- 目标 SigNoz Dashboard 的 MVP 图表无需修改查询字段即可显示 opencode 数据。
- 抽查默认 Profile 和 Claude Code Profile，均无 prompt、response、tool input/output 明文。
- 每个 `prompt.id` 能关联对应的 user prompt、API 和工具事件；存在 Trace 时可继续关联到相同 trace。

## 待确认问题

- 首个兼容基线是否固定为调研时的 Claude Code `2.1.209`，还是只声明文档访问日期。
- opencode reasoning tokens 在 Claude Code Profile 中应仅保留在 `opencode.*` 扩展信号，还是完全不输出。
- 首个版本是否需要兼容某个指定 SigNoz Dashboard JSON；验收前需要获得该 Dashboard 或查询清单。
