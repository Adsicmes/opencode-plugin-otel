# Claude Code 遥测兼容验收标准

## Profile 与回退

1. 未配置兼容 Profile 时，现有 Metric、Event、Trace 名称和计数语义保持不变；工具输入输出明文被移除，并在发布说明中列为安全性行为变化。
2. 通过 plugin option 或 `OPENCODE_*` 环境变量启用 `claude-code` Profile 后，三类信号同时使用兼容映射。
3. option、环境变量和默认值的优先级有自动化测试。
4. 无效 Profile 回退到 `opencode` 并产生不含敏感信息的 warning；环境变量和 plugin option 使用相同规则。

## Metrics

1. fixture 中一次会话、一次成功模型请求、一次 diff 和一次 commit 产生预期的五类 Claude Code Metric。
2. `token.usage` 只使用官方四类 `type`；reasoning 不计入 output，也不映射为不存在的 Claude Code type。
3. Token 和 cost 不因多个事件处理路径重复累计。
4. Claude Code 命名空间中不出现 `cache.count`、`session.duration`、`message.count`、`model.usage`、`retry.count`、`subtask.count` 等 opencode 特有 Metric。
5. 每个 Metric 的 instrument 类型、unit、属性名和枚举与兼容矩阵一致。
6. Claude Code Profile 未显式配置 temporality 时导出 `delta`；显式覆盖时启动日志报告兼容性降级。

## Events

1. 核心事件 body 分别为完整的 `claude_code.user_prompt`、`claude_code.api_request`、`claude_code.api_error`、`claude_code.tool_result` 和 `claude_code.tool_decision`。
2. 每个事件的 `event.name` 不带 `claude_code.` 前缀。
3. 同一 session 中 `event.sequence` 从确定起点单调递增，不重复、不倒退。
4. 一次用户 turn 下的核心事件共享一个 UUID v4 `prompt.id`；下一次用户 turn 使用新 ID。
5. 同一工具调用的 result 和 tool Span 共享 `tool_use_id`；decision 优先使用上游 call ID，缺失时使用随机 UUID。
6. API 成功事件包含可获得的模型、成本、耗时和四类 Token；失败事件包含有界错误摘要和耗时。
7. 缺少上游数据时字段缺省，不输出伪造占位身份数据。
8. permission response 的 source 映射只产生可证明的 `user_permanent`、`user_temporary` 或 `user_reject`，不能把未知来源标成 `config`、`hook` 或 `user_abort`。

## Traces

1. 一次普通交互至少形成 `interaction` 根 Span 及其 `llm_request` 子 Span。
2. 包含工具调用时形成固定名称的 `tool` 子 Span，不把工具名拼进 Span name。
3. Span 包含 `span.type` 和相应核心字段。
4. API 或工具失败时相应 Span status 为 `ERROR`。
5. 能获取上下文的核心 LogRecord 带匹配的 `trace_id` 和 `span_id`。
6. 没有可靠父级时 Span 保持真实根/缺省关系，不构造虚假父级。

## 隐私与身份

1. 默认配置下，导出的 Log、Span attributes 和 Span events 中不出现 prompt、assistant response、tool input、tool output、文件正文或完整命令明文。
2. Resource `service.name` 默认是 `opencode`，scope 属于本项目命名空间；Claude Code Profile 的 `service.version` 和 scope version 是插件真实版本，`app.version` 不伪装为 Claude Code 版本。
3. 输出中不存在插件生成的 `user.email`、`user.account_id`、`user.account_uuid` 或 `organization.id`。
4. 插件只上报自身真实版本，不上报 Claude Code 版本。
5. 兼容 Profile 带有明确的 Schema 兼容标记，消费者可以区分“Claude Code Schema 兼容数据”和“Claude Code 官方生产数据”。

## 生命周期与资源

1. session idle 后清理活动 tool、permission 和 Span 状态，但保留 session sequence 与晚到事件所需的有界关联；session deleted/error 后清理全部关联状态。
2. 所有新增 Map 有大小上限；现有 bounded map 规则不回退。
3. exporter 失败和 handler 异常不影响 opencode 主流程。

## 验证门槛

- `bun run typecheck` 通过。
- `bun test` 通过。
- 新增 Profile 契约测试和默认 Profile 回归测试通过。
- 使用本地 Collector 捕获一次完整 fixture，规范化后与预期快照一致。
- 在目标 SigNoz 实例验证至少 Token、成本、会话、API 错误和 Trace 层级查询。
- `docctl validate` 与 `git diff --check` 通过。
