# Claude Code 遥测兼容技术设计

## 背景与约束

当前实现直接在 handlers 中构造 Metric attributes、LogRecord 和 Span。`metricPrefix` 只影响 Metric 和 Trace 名称，无法表达一整套 Schema 映射。兼容能力必须覆盖三类信号，同时保持默认行为和现有配置兼容。

设计遵循以下约束：

- handlers 继续通过 `HandlerContext` 获取 OTel 与状态，不直接依赖全局对象。
- Token 和 cost 只在 `message.updated` 累计。
- pending tool 和 permission 状态继续有界。
- 不引入 `sdk-node`。
- 不伪造 opencode 没有的身份与事件。
- 默认内容安全性必须先于 Schema 对齐。

## 设计方案

### 兼容 Profile

新增配置：

| PluginConfig / option | 环境变量 | 默认值 | 值 |
| --- | --- | --- | --- |
| `telemetryProfile` | `OPENCODE_TELEMETRY_PROFILE` | `opencode` | `opencode`、`claude-code` |

Profile 是完整遥测契约选择器，不是名称别名。`metricPrefix` 保留给 `opencode` Profile；在 `claude-code` Profile 中，标准 Metric 名由 Schema 固定，用户自定义前缀不覆盖标准名。若需要保留扩展信号，它们必须继续使用 `opencode.` 命名空间。

### 集中式 Schema 描述

增加一个轻量 Schema/Profile 模块，集中提供：

- Metric 名、unit、description 和是否在当前 Profile 启用。
- Event body 与 `event.name`。
- Span name、`span.type` 和 Span kind。
- 属性名和枚举映射。
- 内容字段是否允许输出。

handlers 调用 Profile 提供的名称和映射，不在每个分支散落 `if (profile === ...)`。Profile 不负责读取 opencode 事件或持有运行时状态。

建议类型边界：

```text
TelemetryProfile
├── metrics
├── events
├── spans
├── mapTokenType()
├── mapPermissionSource()
└── contentPolicy
```

### 关联状态

在 `HandlerContext` 增加：

| 状态 | Key | Value | 生命周期 |
| --- | --- | --- | --- |
| `promptContexts` | session ID / run ID | prompt ID、interaction sequence、run ID | 新 user prompt 替换当前指针；deleted/error 删除 run 关联 |
| `eventSequences` | session ID | 下一个 sequence | 每次事件增加；session deleted/error 删除 |
| tool correlation | session ID + call ID | 已有 tool span/pending 数据扩展 `tool_use_id` | tool 完成或 session 结束删除 |

`prompt.id` 在 `chat.message` 收到真实用户 turn 时生成 UUID v4，并沿现有 run/session 关系传递。所有事件从统一 helper 获取公共事件属性，确保 `event.timestamp`、`event.sequence`、`session.id` 和 `prompt.id` 一致。

### Metrics 映射

| 当前信号 | Claude Code Profile | 处理 |
| --- | --- | --- |
| `session.count` | `claude_code.session.count` | 保留；当前不能可靠区分 fresh/resume/continue 时省略 `start_type`，不增加非官方枚举 |
| `token.usage` input/output/cache | 同名标准 Metric | 保留并对齐 type |
| reasoning token | 无标准 type | 不进入 Claude Code Metric；可在 `opencode.token.usage` 扩展输出，MVP 默认关闭扩展 |
| `cost.usage` | `claude_code.cost.usage` | 保留真实成本、模型和可证明的 agent 归因 |
| `lines_of_code.count` | 同名标准 Metric | 保留，但兼容矩阵标注 gross churn 差异；当前 diff 事件无可靠模型归因时省略 `model` |
| `commit.count` | 同名标准 Metric | 保留 |
| opencode-only Metrics | 无等价物 | 不进入 `claude_code.*` |

Instrument 创建必须按 Profile 决定，避免创建但永不使用的大量标准/扩展 instrument。现有 disabled metrics 配置仍按 suffix 处理，并明确作用于 Profile 的逻辑 Metric。Claude Code Profile 在未显式配置 `metricsTemporality` 时注入 `delta`；显式值优先，但非 `delta` 值触发兼容性降级 warning。

### Event 映射

建立统一 `emitEvent` helper，输入逻辑事件名、session、timestamp、attributes 和可选 OTel context，输出 Profile 对应 LogRecord：

```text
emitEvent("api_request", ...)
opencode profile    -> body: api_request
claude-code profile -> body: claude_code.api_request
```

Claude Code Profile 自动添加：

- `event.name`
- `event.timestamp`
- `event.sequence`
- `session.id`
- 当前交互的 `prompt.id`（存在时）
- `telemetry.schema.name=claude-code`

事件特有字段仍由 handler 提供。`success` 等布尔值保持 OTel 原生类型；不得为了匹配某个后端展平视图统一转为字符串。

### Trace 映射

| 当前概念 | opencode Profile | Claude Code Profile |
| --- | --- | --- |
| user turn | `<prefix>session` | `claude_code.interaction` |
| model request | `<prefix>llm` | `claude_code.llm_request` |
| tool call | `<prefix>tool.<name>` | `claude_code.tool` |

Claude Code Profile 的主要属性：

- interaction：`span.type=interaction`、`session.id`、`user_prompt_length`、`interaction.sequence`、`interaction.duration_ms`。
- llm request：`span.type=llm_request`、`model`、`gen_ai.system`、`gen_ai.request.model`、`duration_ms`、Token、attempt、success 和 error/status。
- tool：`span.type=tool`、`tool_name`、`tool_use_id`、duration 和 result size。

OpenInference 属性可以作为真实的扩展属性保留，但不得替代 Claude Code 核心字段。Claude Code Profile 下 tool input/output 默认不写入 `TOOL_PARAMETERS`、`INPUT_VALUE`、`OUTPUT_VALUE`。

### Log 与 Trace 关联

事件产生时若已有对应 Span context，`emitEvent` 将其写入 `LogRecord.context`，由 OTel Logs SDK 解析并写入 `trace_id` 和 `span_id`。由于部分事件在 Span 结束后发出，handler 必须在结束前保留对应 `SpanContext`，再通过 `trace.setSpanContext` 构造日志 context；事件早于 Span 创建或处于会话级时允许没有 trace context。

关联优先级：

1. `trace_id` / `span_id`
2. `tool_use_id`
3. `prompt.id`
4. `session.id` + `event.sequence`

实现测试必须验证实际 exporter 输出，而不仅检查 handler 输入对象。

## 隐私设计

### 默认策略

两个 Profile 都采用内容最小化默认值。这会改变当前 `opencode` Profile 中工具内容可见的行为，属于有意的安全加固；兼容承诺只覆盖信号名称、配置和计数语义，发布说明必须明确该变化。

- prompt 和 response 不写入 Log 或 Span；如契约要求字段存在则使用统一 `<REDACTED>`。
- tool parameters/input/output 不进入 Span 和 Event。
- 仅保留长度、字节数、工具名、模型、耗时、Token、成本和状态。
- error 使用 `errorSummary`，并增加长度上限与敏感模式清理测试。

由于当前实现会把 tool input/output 写入 Span，MVP 必须先修复这一行为，再宣称兼容 Claude Code 默认隐私模型。

### 生产者身份

Resource 和 scope 不参与 Schema 伪装：

- `service.name=opencode`，除非用户通过 Resource 配置显式覆盖。
- Claude Code Profile 设置 `service.version=<插件真实版本>`，并将同一版本传给 Meter、Logger 和 Tracer scope；不设置表示 Claude Code 客户端版本的 `app.version`。`opencode` Profile 暂时保留现有 `app.version` 以避免无关 Schema 变化。
- scope 使用 `com.opencode` 或更具体的本项目 scope。
- 增加 `telemetry.schema.name=claude-code` 和 `telemetry.schema.baseline=<基线>`。

## 兼容性与迁移

- 默认 `opencode` Profile 确保无配置升级不改变信号名称、配置和计数语义；默认移除工具内容是独立记录的安全加固。
- 现有 `OPENCODE_METRIC_PREFIX=claude_code.` 继续工作，但 README 标记为“仅名称前缀兼容”，推荐迁移到 Profile。
- 不自动把已有用户切换到新 Profile。
- Profile 可在重启 opencode 后回退；不涉及持久化数据迁移。
- 同一 service 在同一时间窗口混用两种 Profile 会形成不同 Schema，运维文档应要求用 Resource attribute 标记部署组。

## 失败模式

| 失败 | 行为 |
| --- | --- |
| 无效 Profile | 统一回退到 `opencode` 并记录不含原始配置内容的 warning |
| 找不到当前 prompt context | 省略 `prompt.id`，不生成随机孤立 ID |
| 找不到 tool permission call ID | tool result 正常输出；decision 使用随机 UUID 作为不透明 `tool_use_id`，不冒充真实调用关联 |
| 无法建立 Span 父级 | 创建真实 root Span 或省略 Span，不伪造 parent |
| exporter 失败 | 维持现有异步失败隔离和 shutdown flush 行为 |
| Schema 新版增加字段 | 当前基线保持稳定，通过兼容矩阵和后续版本升级 |

## 测试策略

### 单元测试

- Profile 配置解析和优先级。
- 名称、unit、枚举与内容策略映射。
- prompt ID 生命周期和 event sequence。
- reasoning token、权限 source 和缺失字段处理。
- session cleanup 和 bounded map。

### Handler 契约测试

- 对相同 opencode fixture 分别运行两个 Profile。
- 捕获规范化 Metric measurement、LogRecord 和 Span。
- 删除随机 ID、时间和 batch 顺序后，与契约快照比较。
- 明确断言敏感字段不存在或为 `<REDACTED>`。

### 端到端验证

- 使用本地 OTel Collector 接收 OTLP。
- 执行 user prompt、成功/失败 API、工具、权限、diff 和 commit 场景。
- 在 SigNoz 验证 Dashboard 查询与 Trace/Log 关联。
- 将实测差异写入后续验证报告，不反向修改事实掩盖差异。

## 回滚

将 `telemetryProfile` 改回 `opencode` 并重启 opencode 即可回滚。实现不改变 OTLP endpoint、鉴权、存储 Schema 或已写入的历史数据。若发布后发现隐私问题，应优先禁用 Traces/Logs 或整个 telemetry，再回退版本。
